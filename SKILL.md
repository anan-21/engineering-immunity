# 工程免疫 (Engineering Immunity)

## 核心问题

AI 辅助开发（Vibe Coding）的三大痛点：

1. **MVP 快但加功能就崩** — 新功能引入 bug，越改越多
2. **重复犯错** — AI 在同一类问题上反复出错
3. **上下文丢失** — 项目周期长了，AI 忘记之前的逻辑

## 方法论总览

```
第一轨：3 步开发流程（先收敛 → 再发散 → 再收敛）
第二轨：3 个版本防护（开发版 → 测试版 → 稳定版）
第三轨：bug 沉淀机制（错误 → 规则 → 免疫）
```

## 项目隔离（必须遵守）

本 Skill 只提供通用流程，**不保存、不共享任何项目的 Bug 规则**。

1. 开始任务时，先用 `git rev-parse --show-toplevel` 确定当前项目根目录；不在 Git 仓库时，要求用户明确项目根目录。
2. 只读取和写入 `<项目根目录>/RULES-BUGS.md`。项目不存在该文件时，在该项目根目录创建它。
3. 禁止读取、复制、合并或写入其他项目的 `RULES-BUGS.md`，包括相邻目录、历史项目和 Skill 安装目录。
4. 禁止把项目 Bug 条目写入本 Skill 的 `SKILL.md`；本 Skill 永远只保留通用流程。
5. 切换项目后，丢弃上一个项目的 Bug 上下文，并重新定位、读取新项目自己的规则文件。
6. 每次任务先读本 Skill 同目录的 `GLOBAL-IMMUNITY.md`，再读当前项目的 `RULES-BUGS.md`。前者只存跨项目通用规则，后者只存项目专属规则。

### 全局免疫规则的自动升级

修复 Bug 后，始终先记录到当前项目的 `RULES-BUGS.md`。同时满足以下任一条件时，自动将**抽象后的预防规则**追加到本 Skill 同目录的 `GLOBAL-IMMUNITY.md`：

- 同一根因已在两个或以上相互独立的项目中出现；
- 已由框架、运行时、Git、测试工具或通用工程实践明确证明为跨项目问题。

禁止将业务规则、产品偏好、单一项目配置、未复现猜测或项目路径写入全局文件。全局条目必须包含根因、适用范围、预防规则和升级依据；不要复制项目 Bug 的完整细节。

**写入前提示用户**：确认满足升级条件后、追加到 `GLOBAL-IMMUNITY.md` 前，先向用户说明“发现可跨项目复用的免疫规则”，并展示标题、适用范围、升级依据和预防规则。默认将这作为进度提示后写入；用户要求先审核、修改或拒绝时，等待其决定且不要写入。

项目的技术、UI、功能、接口和数据库规则也应放在同一项目根目录下的 `RULES-*.md`，并且仅对该项目生效。

## 第一轨：3 步开发流程

### 第 1 步：定规则（收敛）

**在做任何功能之前，必须先让 AI 整理并输出以下规则文件：**

#### 1.1 技术栈规则 (`RULES-TECH.md`)
- 明确语言、框架、版本
- 数据库选型、ORM、缓存策略
- 部署环境、CI/CD 要求

#### 1.2 UI/UX 规范 (`RULES-UI.md`)
- 统一的设计风格（颜色、字体、间距）
- 组件库、图标库
- 交互逻辑（loading / empty / error 状态如何展示）
- 响应式断点

#### 1.3 功能规格 (`RULES-FEATURES.md`)
- 每个功能模块的边界定义
- 用户故事和验收标准
- "做到什么程度算完成"的明确标准

#### 1.4 接口规范 (`RULES-API.md`)
- 命名规范（RESTful / GraphQL / RPC）
- 请求/响应格式模板
- 错误码统一规范
- 分页、排序、筛选的通用参数

#### 1.5 数据库规范 (`RULES-DB.md`)
- 命名规范（表名、字段名、索引名）
- 字段类型选择原则
- 索引策略
- 迁移文件管理方式

> **关键原则：规则越清楚，AI 跑偏概率越低。这一步花的时间，后面十倍赚回来。**

### 第 2 步：做模块（发散）

**按功能独立分支开发，但始终在完整工程中联调。每个功能遵循“开发 → 工程级验证 → 通过 → 进入测试版”的闭环：**

1. 将需求拆成可独立交付的功能（登录、注册、首页等），从 `develop` 切出对应的 `feat/*` 分支
2. 开发前列出影响范围：前后端、接口契约、数据库迁移、共享组件、配置和已有用户流程
3. 在同一个功能分支内完成所有必要改动，并使用完整工程运行和联调；“独立”指分支隔离，不是脱离工程开发
4. 运行功能测试，以及受影响已有流程的回归测试、构建和迁移验证
5. 验证通过后，**先合入测试版 `develop`**，不要直接进稳定版；`develop` 负责多功能集成、测试环境和端到端验收
6. 同一时间优先聚焦一个功能；并行开发时，每个功能必须独立分支并提前声明共享改动

**遇到 bug 时的处理流程（不硬改）：**
1. 复现问题并保留当前现场，先不要连续叠加补丁
2. 用测试、日志和 `git diff` 定位根因及受影响范围
3. 修复后补回归测试，并把根因和预防规则沉淀到当前项目的 `RULES-BUGS.md`（见第三轨）
4. 仅在当前方案确认不成立、或需要立即恢复可用环境时，才回滚到上一个可运行版本，并基于新规则重新实现

### 第 3 步：做组装（收敛）

**所有模块在测试版验证通过后，按接口规范组装：**

1. 按 `RULES-API.md` 定义的接口进行模块对接
2. 做端到端集成测试
3. 全部通过后，合入稳定版
4. 打 tag，记录版本号

## 第二轨：3 个版本防护

### Git 分支策略

```
main (稳定版)
  ├── develop (测试版)
  │     ├── feat/user-login (开发版)
  │     ├── feat/user-register (开发版)
  │     └── feat/home-page (开发版)
  └── hotfix/* (紧急修复，直接从 main 切出)
```

### 版本定义

| 版本 | 分支 | 用途 | 准入条件 |
|------|------|------|---------|
| **稳定版** | `main` | 可随时部署的生产代码 | 测试版全部模块通过 + 集成测试通过 |
| **测试版** | `develop` | 已验证的模块集成区 | 单个模块测试通过 + 代码审查通过 |
| **开发版** | `feat/*` | 你和 AI 实际写代码的地方 | 无限制，允许试错 |

### 版本回滚策略

```
开发版出问题 → 在当前 feat 分支复现、定位和修复；当前方案确认不成立时，才回到 develop 重新切分支
测试版出问题 → 从 develop 切 fix 分支修复；需要恢复测试环境时，先部署上一个可用版本，禁止未保存就回退 develop
稳定版出问题 → 紧急！切 hotfix 分支修复，打补丁版本
```

### 关键原则

- **绝不直接在 main 上改代码**
- **新功能必须经过 开发版 → 测试版 → 稳定版 的完整链路**
- **每次合入测试版之前，让 AI 跑一遍已有测试，确保没有回归**
- **回滚优先于硬改** — 不知道哪里出问题时，先回滚再排查

## 第三轨：Bug 沉淀机制

### 规则文件 (`RULES-BUGS.md`)

每次修复 bug 后，AI 必须把以下信息写入**当前项目根目录**的 `RULES-BUGS.md`：

```markdown
### Bug #001: [简短标题]

- **现象**：[用户看到什么]
- **根因**：[技术层面什么原因]
- **修复**：[怎么改的]
- **预防规则**：[下次 AI 要遵守什么规则才能避免]
- **关联模块**：[影响的功能模块]
```

符合“全局免疫规则的自动升级”条件时，同时在 `GLOBAL-IMMUNITY.md` 追加：

```markdown
### Global Rule #001: [简短标题]

- **根因**：[可跨项目复用的技术原因]
- **适用范围**：[语言 / 框架 / 工程场景]
- **预防规则**：[未来任务必须遵守的约束]
- **升级依据**：[多项目复现或权威技术依据]
```

### 沉淀原则

1. **每个 bug 必须沉淀**，不能修完就走
2. **预防规则要写成 AI 能理解的约束**，比如"在所有表单组件中，必须处理 loading / error / empty 三种状态"
3. **新功能开发前，AI 必须先读当前项目的 `RULES-BUGS.md`**，检查是否有相关预防规则；禁止读取其他项目的记录
4. 规则文件随项目迭代持续更新，是项目的"免疫系统"

## 操作流程速查

### 启动新项目

```
1. 创建 RULES-TECH.md, RULES-UI.md, RULES-FEATURES.md, RULES-API.md, RULES-DB.md
2. git init → 创建 main 分支（稳定版）
3. git checkout -b develop（测试版）
4. 对每个功能模块：
   a. git checkout -b feat/xxx（开发版）
   b. 让 AI 读规则文件 → 实现功能 → 写测试 → 跑测试
   c. 测试通过 → 合入 develop
   d. 回到步骤 a 做下一个模块
5. 所有模块完成 → develop 合入 main → 打 tag
```

### 修复 bug

```
1. 确认 bug 出现在哪个版本
2. 如果在稳定版 → 从 main 切 hotfix 分支
   如果在测试版 → 从 develop 切 fix 分支修复，禁止直接修改 develop
   如果在开发版 → 在当前 feat 分支修复
3. 修复 → 测试 → 沉淀到 RULES-BUGS.md
4. 合入对应分支
```

### 添加新功能（项目中期）

```
1. 先让 AI 读取所有 RULES-*.md 和 RULES-BUGS.md
2. 从 develop 切出 feat/xxx
3. 开发 → 测试 → 沉淀（如有新 bug 规则）
4. 合入 develop
```

## 卡住检测与自动熔断机制

### 什么是"卡住"

当满足以下任一条件时，说明当前任务已经陷入死循环：

| 信号 | 判定标准 |
|------|---------|
| **反复失败** | 同一任务连续 3 次尝试仍未解决 |
| **来回修** | 修 A 坏 B，修 B 坏 A，来回 2 轮以上 |
| **补丁越打越多** | bug 修复引入了新 bug，新 bug 的修复又引入更新的 bug |
| **用户反复说不** | 用户连续 2 次以上说"不对""还是有问题""又出错了" |

### AI 检测到卡住时的行为（自动触发）

AI 检测到上述信号后，**必须停止继续修改**，执行以下流程：

```
1. 停止：立即停下来，不要再改了
2. 告知：告诉用户"当前任务已连续 X 次未通过，建议回滚排查"
3. 建议：列出下一步选项让用户选择：
   a. 回滚到上一个可用版本，对比变更，找到元凶
   b. 把当前 bug 先记录到 RULES-BUGS.md，换一条思路
   c. 缩小范围：把任务拆成更小的步骤，逐步验证
4. 等待用户选择后执行
```

### 卡住时的排查流程

```
已确认卡住
    │
    ▼
git log --oneline -5               # 找最近一次能运行的 commit
    │
    ▼
git diff <good-commit> HEAD        # 对比所有变更
    │
    ▼
定位到具体改动后:
  · 这个改动为什么没解决问题？
  · 有没有更简单的方式达到同样目的？
  · 是不是任务目标本身就不合理？
    │
    ▼
把分析结果写入 RULES-BUGS.md → 回到好版本 → 基于新规则重做
```

### 关键原则

- **回滚是工具，不是失败** — 回滚不代表做错了，代表你在用正确的方式解决问题
- **硬改是最大的时间杀手** — "再试一次"比"回滚重做"通常花更多时间
- **补丁超过 3 个就该重写** — 如果一段代码因为同一个功能被改了 3 次以上，说明第一次的方案就有问题

## 语言支持

- 支持中文和英文提示词；用户使用哪种语言，就用哪种语言沟通、输出计划和提示全局规则升级。
- 规则文件名、分支名和字段结构保持不变：`RULES-*.md`、`GLOBAL-IMMUNITY.md`、`feat/*`、`fix/*`、`hotfix/*`。
- 英文项目可以用英文撰写规则内容；不创建按语言重复的规则文件。

## 使用本 Skill

在 Claude Code 中，你可以通过以下方式调用这个工作流：

- **开始新功能**：告诉 AI "按照工程免疫，开始一个新功能：[功能描述]。先检查规则文件。"
- **修复 bug**：告诉 AI "按照工程免疫修复这个 bug：[bug 描述]。修复后沉淀规则。"
- **版本回滚**：告诉 AI "当前开发版有问题，回滚到测试版重新来。"
- **初始化项目**：告诉 AI "按照工程免疫初始化这个项目，先生成所有规则文件。"
- **卡住了**：告诉 AI "又报错了，这是第 3 次了"、或者直接说"卡住了，按三轨流程排查"——AI 会自动执行卡住检测流程。

> **提示**：AI 现在也会主动监测卡住信号（反复失败、来回修、补丁叠补丁），检测到时会主动建议回滚，不给用户提需要知道这些命令。

### English prompts

- **Start a feature**: "Use Engineering Immunity to start [feature]. Read the current project's rules and bug history first."
- **Test a feature**: "Use Engineering Immunity to test [feature], including affected regression tests. Promote it to the integration branch only if it passes."
- **Fix a bug**: "Use Engineering Immunity to fix [bug]. Reproduce it, identify the root cause, add a regression test, and record the prevention rule."
- **Release**: "Use Engineering Immunity to release [feature]. Verify integration-environment acceptance before promoting it to the stable branch."
- **Stuck**: "We're stuck. Follow the Engineering Immunity circuit-breaker process."
